Lateral File System (LAFS)

VirtualFileSystem

Goals

  • Reduce read operations required by storage volume to traverse filesystem structure (i.e, lookup file contained in several layers of filesystem)
  • Allow easy prioritizing of filesystem caching (isolate specific directory structures which are more frequently accessed and optimize this portion in a memory cache)
  • Remain conscious of DCPU's memory limitations and optimize process of caching with this in mind
  • Allow smaller filesystems to operate at a highly optimal rate without sacrificing scalability
  • Allow filesystem to remain as predictable and contiguous as possible.

Advantages

  • Due to the directory entry structure, file data can overlap directory data. That is to say, a directory entry can describe both a directory and further describe the structure of a file. Allowing for more efficient use of space.
  • All entries have a known sector allocated for them and are very predictable in nature.
  • Directory flats can be very easily placed into memory.
  • 128 entries can be cached and easily traversed with an allocation of only one 512 word buffer. Small filesystems will benifit greatly from this. Larger filesystems will need additional memory buffers to maintain look-up times.
  • Efficient on filesystems where files are under 3 or 4 sectors in length
  • Quick to search filesystem due to string flat arrangement.

Disadvantages

  • Due to tight size restrictions, the size of a file is only accurate to a resolution of 2 words (4 bytes). While this does not waste space per se, it forces files to align their size to 2 words (i.e, this does not force files to overflow onto another sector.)
  • There is no room for metadata, such as user access permissions or date the file was created accessed.
  • Support is limited to strictly two types of entries, directories or files. There is no support for symbolic links or more advanced structures present in the FAT16 fs.
  • Due to the limitation in length of the extended next entry (6bits) file extentions cannot be further than 63 sectors apart.
  • Less efficient when majority of files exceed 5 or 6 sectors in length. Efficiency will decay with many larger files
  • Names limited to single byte ascii, with a length of 6 characters.

Filesystem structure

The filesystem is laid out as follows

Name Size (Words) Description
Boot Code 512 Contains code for the first stage bootloader.
Super Block 512 Contains basic information about filesystem, and bitmaps for Directory Descriptor Flats
First Directory Descriptor Flat 128*512 + 512 First directory flat in filesystem
... ... ...
Last Directory Descriptor Flat 128*512 + 512 Last directory flat in filesystem

String Flat

Is a series of 6 byte (3 word) single byte ascii strings, padded with null. A string flat is placed with a series of directory entries, and describes the name of each directory entry in parallel. I.e, entry 0 in the directory list will have a name which corresponds to the first entry of the string flat which accompanies it.

Directory Descriptor Flat

A directory descriptor flat is a portion on the storage consiting of a series of directory entries proceeded with what is called a string flat, which describes, in parallel, the name of each directory entry. The size of a directory flat and its string flat must equal to the size of a single sector.

Solving for the system: x + 3x = 512, defines a total of 128 directory entries per a directory flat.

The number of directory flats present in a file system is to be determined when the volume is formatted and placed in the super block. Each directory flat is followed by 128 adjacent sectors allocated for files in the flat.

Structure of Entries

All entries are organized in a linked list. Each entry can describe a directory, a file, or both a directory and an extention to a file. File extentions are described later.

Each directory entry corresponds to a name in the String Flat at an index that is the equal to the index of the entry. Further, the entry also corresponds to (i.e, owns) one of the 128 sectors which follows the File Descriptor Flat. The sector this entry owns has an index equal to the index of the entry. That is to say, the entry at index 'x', has its name described in the string flat at index 'x', and owns the sector that is 'x' sectors after the Directory Descriptor Flat. Hence the name "Lateral Filesystem."

Directory structures (folders, subfolders etc...) are described using a linked list. When an entry of type "directory" is encountered while traversing the linked list of entries, all entries there after describe the files which belong to that folder. If, while traversing the linked list, yet another entry of type "directory" is encountered, all following entries belong to this new folder (which is contained inside of the previously encountered directory.) Directories are terminated with an entry of type "directory" where the TerminateList bit is set.

If a directory contains many files, it can be time consuming to traverse the list to the end of the directory. To avoid wasting time, one may place a "Skip Directory" Skip directories function as any normal directory (i.e, they also link to the next file in the chain) the difference is in their name. A skip directory's name will look like this:

Word # Value Meaning
0 ".\0" A period followed by a null terminator. This means that this directory is a skip directory
1 AfterEndNext This links to the entry that is just outside of the directory
2 Undefined Reserved

A skip directory can be used to skip a directory if the file being looked for is known to not be located inside of that directory.

An example of this structure is illustrated below using an ascii representation of the file entries (F is file, { is new folder, } is end of folder and ^ is a "Skip Directory Entry")

String Flat:
"file01" "file02" "fold1" "fold2" "file inside of fold1\fold2\" NullName NullName
F1 F2 { ^ { F3 } F4 }

The ^ will have its name (on order of 0th word to 2nd word) as (".\0", [Offset from current entry to F4], reserved)

Also it is critical to note that the directory which terminates the list by having the TerminateList bit set may also be used to describe a new directory, and may also be allocated as a file extention. Otherwise it may simply mark the end of a list of directories. Note though, however, that these entries (those which terminate folders) along with all which define the start of a folder can still be allocated as a file extention as not to waste the sector they own (more on file extentions later)

This folder structure allows folders to be very easily and quickly appended to folders of any layer in the filesystem.

Directory entries of type "file" work exactly as entries which describe folders in the sense that they own a sector and have a corresponding entry in the String Flat with an index equal to their own. Also, like directory entries, each file entry links to another entry to describe the next file in the system. However, the structure of a file entry differs.

If a file is greater than one sector(I.e it consumes more than the one sector it owns), it must be linked (with the next member) to a file extention entry. Folders can act as both folders and file extention entries and so to locate a file extention, one can either scan for folders which have not yet been allocated as an extentions, or allocate a new entry and use it as a file extention (and if need be later, a folder as well)

Data Structures

Boot-Stage 1 Sector

The first sector of the storage media is reserved for the first-stage bootloader.

Super-Block

This is the second sector of the storage media and is reserved for the super block. Immediately following this sector are the Directory Descriptor Flats. The first one describing the root directory.

Name Size (words) Meaning
volume name 4 The volume name, padded in null, single byte ascii string.
reserved 4 Reserved
DirDescFlatCount 1 Number of directory descriptor flats
DirDescFlatBitmap 8 Allocation bitmap for first directory descriptor flat
... ... ...
DirDescFlatBitmap 8 Allocation bitmap for last directory descriptor flat

Directory Entry Structure for Directory

The total size of a directory entry is one word

Member Name Size Definition
DirectoryFlag 1 bit 0 = file, 1 = dir
ExtendedNext 6 bit Next entry if accessed as extention. If 0, this dir is not allocated as an extention.
TerminateList 1 bit If set, the directory terminates the containing list.
LongJumpFlag 1 bit Set if NextEntry describes a directory descriptor flat (relative to the current one)
EndExtFlag 1 bit If being accessed as an extention, set if this directory describes the last of a file's extention.
NextEntry 6 bit Offset in words to next entry if not accessed as extention, or, if LongJumpFlag is set, this is the signed integer of Directory Flats to jump relative to this one.

Directory Entry Structure for a File

Member Name Size Definition
DirectoryFlag 1 bit 0 = file, 1 = dir
TrailingSize 8 bit Signed (trailing file size - 1 in magnitude) / 2
ExtentedFlag 1 bit Set if the next entry should be accessed as an extention (i.e, the file is extended)
NextEntry 6 bit Next entry

Design Considerations \ Implementation Process

It is important to note that traversing through a directory can be an exhausting process, specifically if the directory requires a jump to a different directory descriptor flat. Due to this, all jumps should be located at the end of the chain described in the flat. Further, all directories in general should be placed near the end of the chain. Finally, if a directory is long, a skip directory should be placed wherever possible.

Calculating Total Size

Calculating the total size explicitly may not be required, since the trailing size can be used to determined how many bytes of the last sector are owned by the file.

However the total size of a file is as follows:

(1 + number of next entries)*512 + 256 + Trailing Size (noting that trailing size is a signed number)

Also note that the TrailingSize member of a file entry is actually TrailingSizeNOT to get TrailingSize from TrailingSizeNOT:

TrailingSize = (TrailingSizeNOT + 1 in magnitude) * 2

Isolating Busy Directories to get the most of LAFS

One of the key features of LAFS is that it allows you to isolate busy directories of your filesystem. And further, easily cache and thus optimize access to these specific ioslated regions of your filesystem. It is thus encouraged to use long jump directories to isolate the root directory from other busy directories. If your directory structures are small, it may be more optimal to do the reverse (that is, isolate your less busy and potentially larger directories.) This filesystem has yet to be implemented and tweaked, so the specifics on that are not yet known.
